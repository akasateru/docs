# 1. 障害調査の進め方

DevOps領域における障害・不具合調査を、場当たり的にならず効率よく進めるための基本的な手順とチェックポイントをまとめる。

## 1.1. 調査の基本ステップ

1. **事象の確認**: いつから・どの範囲で・どんな症状が起きているかを具体的に把握する（再現手順、影響範囲、発生時刻）。
2. **影響度の判断**: ユーザー影響の有無・規模から対応優先度を決める。
3. **仮説立案**: 直近のデプロイ、設定変更、外部サービス障害など、変化点を軸に仮説を立てる。
4. **検証**: ログ・メトリクス・トレースを使って仮説を裏付ける、または棄却する。
5. **暫定対応と恒久対応の切り分け**: まずサービス影響を止める応急処置を優先し、根本原因の修正は別途対応する。

## 1.2. よく見るべき情報源

| 情報源 | 確認できること |
| ----------- | --------------------------------------------- |
| アプリログ | エラーメッセージ、スタックトレース |
| メトリクス | CPU/メモリ使用率、レイテンシ、エラー率の推移 |
| トレース | リクエストがどのサービス・処理で詰まっているか |
| デプロイ履歴 | 直近の変更点との相関 |

## 1.3. 変化点分析（Change Point Analysis）

障害の多くは「何かが変わった」ことに起因する。以下を時系列で並べて事象発生タイミングと突き合わせるのが有効。

- コードのデプロイ
- インフラ構成変更（スケーリング設定、DNS、証明書更新など）
- 外部依存サービスの仕様変更・障害
- トラフィックパターンの変化（急増、特定リクエストの偏りなど）

## 1.4. ポストモーテム（振り返り）

- 個人の責任追及ではなく、仕組みの改善に焦点を当てる（Blameless Postmortem）。
- タイムライン、根本原因、再発防止策を文書化し、チームで共有する。
