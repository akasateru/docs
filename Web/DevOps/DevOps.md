## 2. 割り込みタスク

「急な仕事に対応する」という根性論ではなく、**「いかに割り込みを可視化し、システムの安定性を保ちながら、開発スピードを落とさないか」**という仕組みづくり

### 2.1. 可視化とフローの管理

- **カンバン方式の活用:** 割り込みが入った際、それを付箋やチケットとしてボードに乗せ、現在の作業（WIP: Work In Progress）を制限。
- **テレメトリ（計測）:** どの程度の割合が「予定された作業」で、どの程度が「割り込み」だったかをデータ化し、次回のスプリント計画に反映させる。

### 2.2. 自動化による「割り込み」の最小化

- **CI/CD:** 手動デプロイによるミス（＝緊急の修正依頼）を減らす。
- **セルフサービス化:** 開発者がインフラ設定を自分で行えるようにすることで、「〇〇の設定をお願いします」というインフラ担当者への割り込みを解消。

### 2.3. 割り込みタスクに対処するための「DevOps的」なステップ

| ステップ           | アクション                                                             |
| ------------------ | ---------------------------------------------------------------------- |
| **可視化**         | 全ての割り込みをチケット化し、誰が何をやっているか明確にする。         |
| **トリアージ**     | その割り込みは「今すぐ」やるべきか、バックログに入れるべきか判断する。 |
| **自動化**         | 同じ原因で発生する割り込みを、スクリプトやツールで自動解決する。       |
| **フィードバック** | 割り込みの発生源（バグ、技術負債など）を根本から修正する。             |
