## 1. .wslconfig設定

- Windowsユーザーのフォルダ直下に.wslconfigを作成

```toml
[wsl2]
memory=10GB  # WSL2が使用する最大メモリを制限（PCの搭載メモリの半分程度が目安）
processors=6  # Windows側のプロセッサをいくつ使うか
networkingMode=mirrored  # WindowsとLinuxでIPアドレスを共有するモード
swap=4GB  # スワップファイル（ディスクへの退避領域）のサイズを制限

[experimental]
autoMemoryReclaim=gradual  # 使わなくなったメモリを少しずつWindowsに返却
```

## 2. WSL+VPNでIPアドレスが変わる

### 2.1. 現状整理

登場するサーバー
・Windwos
・WSL
・社内サーバー
・test環境のアプリケーション（urlでアクセス）

条件
・手元のWSLから社内サーバーに向けてVPN接続をする
・test環境はVPN接続するか社内からのみアクセス可能
・リモートからはVPN接続してtest環境にアクセスする

$vpn = Get-NetAdapter -Name "Connect Tunnel"; New-NetRoute -DestinationPrefix "34.111.166.6/32" -InterfaceIndex $vpn.ifIndex -NextHop "0.0.0.0" -RouteMetric 1

変更前

```bash
1. Playwright → GET https://console.test-genai.rtoaster.jp/login

2. サーバー → 307 Temporary Redirect
   Location: http://console.test-genai.rtoaster.jp/login/   ← HTTP で返している

3. Playwright（ブラウザ）がリダイレクトに従い
   → GET http://console.test-genai.rtoaster.jp/login/ へ接続しようとする

4. 接続失敗
   （HTTP では応答しない / すぐ切れる など）
   → net::ERR_EMPTY_RESPONSE や socket hang up

5. ログインページに到達できず、テスト失敗
```

変更後

```bash
1. Playwright → GET https://console.test-genai.rtoaster.jp/login
   （addRedirectHttpsRoute で全リクエストを route がインターセプト）

2. route.fetch({ maxRedirects: 0 }) でサーバーにリクエスト
   → リダイレクトは自動追跡しない（HTTP へ飛ばない）

3. サーバー → 307 + Location: http://... を route が受け取る

4. route 内で Location を https:// に書き換え
   → route.fulfill({ status: 307, headers: { location: "https://..." }, body }) でブラウザに返す

5. ブラウザは「307 + Location: https://...」として受け取り
   → GET https://console.test-genai.rtoaster.jp/login/ へ遷移（HTTPS のまま）

6. 以降も HTTPS のみで通信し、ログインページに到達 → テスト成功
```

## 3. 解決案

### 3.1. 案1:WSL2の「ミラーモード」を使用する

- Windowsの .wslconfigに以下を追記して保存

  - ```bash
        [wsl2]
        networkingMode=mirrored
- PowerShellで再実行： `wsl --shutdown`

### 3.2. 案2:wsl-vpnkit を導入する

Windows 10や、上記ミラーモードが不安定な場合、wsl-vpnkit を使用して、WSL2内のネットワークをVPN経由にする方法が一般的です。これは、VPN経由の通信をルーティングする仮想マシンをWSL上で動かすツールです。

```bash
$vpn = Get-NetAdapter -Name "Connect Tunnel"; New-NetRoute -DestinationPrefix "34.111.166.6/32" -InterfaceIndex $vpn.ifIndex -NextHop "0.0.0.0" -RouteMetric 1
```

## 5. 重要なポイント

### 5.1. ミラーモードの特徴

- **WindowsとWSL2が同じIPアドレス (192.168.1.3) を共有**
- `localhost` (127.0.0.1) がWindowsとWSL2間で共有される
- ポートフォワーディングが不要

### 5.2. ネットワークインターフェース

#### 5.2.1. Windows側

- **Wi-Fi**: 192.168.1.3/24 (アクティブ)
- **Hyper-V仮想アダプタ**: WSL2との通信用
- **SonicWall VPNアダプタ**: 現在未接続
  - VPN接続時は 172.21.80.0/20, 172.26.160.0/20 のルートが追加される

#### 5.2.2. WSL2側

- **eth0**: 192.168.1.3/24 (Windowsと同じIP)
- **lo**: 127.0.0.1, 10.255.255.254 (WSL DNSプロキシ)

### 5.3. ルーティング

#### 5.3.1. Windows側

```
0.0.0.0/0 → 192.168.1.1 (デフォルトゲートウェイ)
127.0.0.0/8 → 127.0.0.1 (ループバック)
192.168.1.0/24 → 192.168.1.3 (ローカルネットワーク)
172.21.80.0/20 → 172.21.80.1 (VPNルート - 未接続時も存在)
172.26.160.0/20 → 172.26.160.1 (VPNルート - 未接続時も存在)
```

#### 5.3.2. WSL2側

```
0.0.0.0/0 → 192.168.1.1 via eth0 (デフォルトゲートウェイ)
192.168.1.0/24 → eth0 (ローカルネットワーク)
192.168.1.1 → eth0 (ゲートウェイ)
```

### 5.4. DNS設定

#### 5.4.1. Windows側

- プライマリDNS: 192.168.1.1
- IPv6 DNS: 240b:11:3e05:1500:569b:49ff:fe3e:ecfc
- ドメイン: brainpad.co.jp, flets-east.jp, iptvf.jp

#### 5.4.2. WSL2側

- nameserver: 10.255.255.254 (WSL DNSプロキシ)
- search: brainpad.co.jp flets-east.jp iptvf.jp

### 5.5. アプリケーション設定

#### 5.5.1. console-api

- **現在**: 127.0.0.1:8080 でリッスン
- **推奨**: 0.0.0.0:8080 でリッスン (ミラーモードでWindows側からアクセス可能)

#### 5.5.2. console-ui

- **設定済み**: 0.0.0.0:3000 でリッスン (nuxt.config.ts)
- Playwrightテストから localhost:3000 でアクセス可能

### 5.6. VPN接続時の注意点

VPN接続時は以下のルートが追加されます：

- 172.21.80.0/20 → 172.21.80.1
- 172.26.160.0/20 → 172.26.160.1

ミラーモードでは、VPN接続時もWindowsとWSL2が同じIPアドレスを共有するため、
VPN経由の通信も両方から可能です。

## 6. トラブルシューティング

### 6.1. localhostにアクセスできない場合

1. サーバーが `0.0.0.0` にバインドされているか確認
2. ファイアウォール設定を確認
3. WSLを再起動: `wsl --shutdown`

### 6.2. VPN接続時の問題

- ミラーモードでは通常問題なし
- 問題が発生する場合は、VPNのIPアドレス範囲とWSL2のサブネットが競合していないか確認
