
## 1. HMAC

- Hash-based Message Authentication Code
- メッセージ認証コード（MAC）の１つ

下記を保証する

1. メールの送信者が第三者のなりすましでないか
2. メッセージが改竄されていないか

## 2. セキュリティ対策

### 2.1. クレデンシャル強化

- シークレットの載せ方
  - URL より Authorization / 専用ヘッダ に寄せる（ログ・Referer 漏えい対策）。
- キーの種類の分離
  - ブラウザ用とサーバー用でキー／スコープを分ける、サーバー用のみ 強い操作を許す、など。
- ローテーション・失効
  - 漏えい時の手順と、定期的な入れ替え。
- レート制限・異常検知
  - キー／IP 単位のスロットリング、監視。

### 2.2. トークンベース

- 短命アクセストークン
  - 長寿命の API キーはバックエンドだけが保持し、外向きは 短命 Bearer のみ、など。
- OAuth 2.0 Client Credentials
  - サーバー間の定番。実装は 認可サーバー＋プロダクト または API Gateway に寄せるパターンが多い。
- JWT の検証
  - 自前または IdP が発行し、署名・有効期限・audience を API 側（またはゲートウェイ）で検証。

### 2.3. ネットワーク・エッジ

- IP 許可（WAF / LB / Cloud Armor 等）
  - 固定出口が取れる場合に有効。テナントごと にやるかは設計次第。
- Private 接続 / 内部 LB
  - インターネットに URL を出さない構成。要件・コストは上がりがち。
- mTLS
  - クライアント証明書で「このサーバーから」まで縛る。運用負荷は高め。

サーバー間連携では、API キー／シークレットの扱いに加え、必要に応じて短命トークン・OAuth（M2M）・IP 制限・BFF／ゲートウェイ・閉域など、ブラウザに依存しない層での対策を組み合わせるのが一般的
