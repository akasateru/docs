# 1. Apache Hadoop

大量のコモディティサーバーを束ねてストレージと計算処理を分散させる基盤。Google のGFS/MapReduce論文（2003〜2004年）がベースになっている。

## 1.1. 全体構造

Hadoopは大きく3つの層で構成される。

| 層               | コンポーネント | 役割                                       |
| ------------------ | ---------------- | --------------------------------------------- |
| ストレージ層     | HDFS              | 大容量データを複数マシンに分散して保存        |
| リソース管理層   | YARN              | クラスタのCPU・メモリをジョブに割り当て       |
| 処理エンジン層   | MapReduce（/ Spark, Tez等） | 分散処理を実行                    |

初期のHadoop（v1）はMapReduceがリソース管理と処理実行を兼ねていたが、v2でYARNとしてリソース管理を分離し、MapReduce以外の処理エンジン（Sparkなど）もYARN上で動かせるようにした。

## 1.2. HDFS（Hadoop Distributed File System）

- ファイルを大きなブロック（デフォルト128MB）に分割し、複数マシンに分散配置する。
- 各ブロックはデフォルト3重に複製（レプリケーション）され、マシン故障してもデータが失われない。
- **NameNode**: どのファイルがどのブロックで構成され、どこに配置されているかという「目次（メタデータ）」を管理する司令塔（1台、またはHA構成で冗長化）。
- **DataNode**: 実際のブロックデータを保持し、読み書きを行う作業員（多数）。

## 1.3. YARN（Yet Another Resource Negotiator）

- クラスタ内のCPU・メモリをどのジョブにどれだけ割り当てるかを管理する。
- **ResourceManager**: クラスタ全体のリソース采配を行う司令塔（1台）。
- **NodeManager**: 各マシン上でリソース使用状況を報告し、実際にタスクを実行（多数）。

## 1.4. MapReduce

- 「Map（データを分割して個別に変換）→ Shuffle（キーごとに集約）→ Reduce（集計）」の2段階で分散処理を記述するプログラミングモデル。
- 例: 大量のログから単語出現回数を数える場合、Mapで各行を`(単語, 1)`のペアに変換し、Reduceで同じ単語ごとに合計する。
- 記述の生産性が低いため、実務ではHiveなど上位レイヤーのツール経由で使われることが多い。

## 1.5. エコシステム

| ツール       | 役割                                                    |
| -------------- | ----------------------------------------------------------- |
| Hive           | SQLライクなクエリでMapReduce/Tezジョブを裏で生成           |
| HBase          | HDFS上に構築されたNoSQL DB（ランダムアクセス向き）          |
| [Spark](Spark.md) | MapReduceよりインメモリで高速な分散処理エンジン（YARN上で動くことも多い） |
| Sqoop / Flume  | RDBMSやログデータをHDFSに取り込むツール                     |

## 1.6. 学習の進め方（次の一歩）

- 手を動かすなら、Docker Composeで疑似分散モードのHadoopを構築し、HDFS操作やMapReduceジョブを実際に動かしてみるのが理解が早い。
