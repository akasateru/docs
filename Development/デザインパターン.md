## 1. デザインパターン

- コーディングにおける様々なパターンをまとめたもの。
- God of 4 と呼ばれる4人によってまとめられた。23個のパターンから構成されている。

### 1.1. デザインパターン3つのカテゴリ

### 1.2. 生成に関するパターン（Creational）

| パターン         | 概要／目的                                                             |
| ---------------- | ---------------------------------------------------------------------- |
| **Singleton      | インスタンスを1つだけに制限。グローバルな状態管理で使用                |
| Factory Method   | サブクラスにインスタンス生成を委ねる。依存を抽象にする                 |
| Abstract Factory | 関連するオブジェクト群をまとめて生成。UIテーマなどで有効               |
| Builder          | 複雑なオブジェクト生成をステップごとに構築。順序や柔軟性が必要なときに |
| Prototype        | 既存オブジェクトを複製して新規生成。コピー機能を使いたいとき           |

### 1.3. 構造に関するパターン（Structural）

| パターン  | 概要／目的                                                           |
| --------- | -------------------------------------------------------------------- |
| Adapter   | インターフェースの食い違いをつなぐ。互換性のないAPIを接続            |
| Bridge    | 実装と抽象を分離して独立に拡張可能にする（例：描画APIとUI部品）      |
| Composite | ツリー構造で再帰的にオブジェクトを扱う（例：フォルダとファイル）     |
| Decorator | 振る舞いをラップして動的に追加（例：ロギング、キャッシュ）           |
| Facade    | サブシステムを簡単なインターフェースで包む。複雑な内部構造を隠す     |
| Flyweight | 多数のインスタンスの共通部分を共有し、メモリを節約                   |
| Proxy     | 実体の代わりに代理オブジェクトを使う（遅延初期化、キャッシュ、認可） |

### 1.4. 振る舞いに関するパターン（Behavioral）

| パターン名                  | 概要と用途                                                                      |
| --------------------------- | ------------------------------------------------------------------------------- |
| Dependency Injection        | コンストラクタなどを使って依存を外部から注入し、結合度を下げる                  |
| Service Locator             | サービス（依存）を登録・取得するハブを作り、依存解決を集中管理                  |
| Repository                  | 永続化のロジック（ORMなど）をドメインロジックから分離                           |
| Unit of Work                | 複数のDB操作を一括で管理・コミット（トランザクション管理）                      |
| Specification               | 条件・ルールの組み合わせ可能なオブジェクト化（柔軟なクエリ構築）                |
| CQRS                        | 読み取り（Query）と書き込み（Command）を分離し、スケーラビリティ向上            |
| Event Sourcing              | 状態変化をイベントとして記録。状態はイベントの合成で再構成                      |
| Publish-Subscribe           | イベント中心で疎結合な通知（非同期含む）。マイクロサービス間通信で多用          |
| Observer/Listenerの非同期版 | イベントバスを介した通知（PubSubやRxなど）                                      |
| Model-View-ViewModel (MVVM) | UIとロジックをデータバインディングで分離（特にフロントエンド）                  |
| Hexagonal Architecture      | ドメインを中心に「入力（Port）」と「外部（Adapter）」で接続。FastAPIなどと相性◎ |
